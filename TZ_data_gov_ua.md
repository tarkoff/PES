# ТЕХНІЧНЕ ЗАВДАННЯ (ТЗ)

## на розробку додатку для імпорту даних (XML, JSON, XLS, XLS, XLSX) з порталу data.gov.ua

---

### 1. Загальні відомості

**1.1. Назва проекту:** Додаток для імпорту та обробки відкритих даних з порталу data.gov.ua.  
**1.2. Мета проекту:** Розробка програмного рішення для автоматизованого або ручного завантаження, парсингу, валідації та збереження наборів відкритих даних (у форматах XML, JSON, XLS) з Єдиного державного веб-порталу відкритих даних (data.gov.ua) для їх подальшого використання, аналізу або відображення в інших підсистемах.

---

### 2. Функціональні вимоги

**2.1. Взаємодія з порталом data.gov.ua (API / Пошук):**

- Підтримка офіційного API CKAN, який використовується на data.gov.ua, для пошуку репозиторіїв та отримання прямих посилань на файли актуальних версій.
- Можливість пошуку наборів даних за ключовими словами, категоріями або ідентифікатором (ID) набору/ресурсу бази data.gov.ua.
- Налаштування підписок: можливість вказати розклад для регулярного та автоматичного оновлення/перезавантаження конкретних наборів даних.

**2.2. Завантаження та імпорт даних:**

- Підтримка розпізнавання та читання форматів: `.xml`, `.json`, `.xls` та `.xlsx`.
- Можливість завантаження великих файлів (понад 1 ГБ) із використанням потокового читання (stream), щоб мінімізувати споживання оперативної пам'яті сервером.
- Можливість завантажувати ZIP архіви та розпаковувати їх, післячого визначити формат розпакованих даних та імпортувати їх.
- Управління чергами: виконання процесу завантаження та імпорту виключно у фоновому режимі (з використанням брокерів повідомлень) з можливістю моніторингу прогресу.

**2.3. Обробка, валідація та нормалізація:**

- **JSON:** Гнучкий мапінг (встановлення відповідностей) вкладених структур JSON у плоский табличний/реляційний вигляд.
- **XML:** Читання деревоподібної структури з можливістю вказати XPath-шлях до основного списку елементів (records).
- **XLS/XLSX:** Вибір конкретного аркуша (sheet), визначення рядка із заголовками (header row) та ігнорування порожніх рядків.
- Валідація: автоматичне розпізнавання і перевірка типів даних (числа, дати, текст).
- Обробка помилок у структурі: механізм пропуску невалідних або пошкоджених записів (skip & log) із занесенням їх у звіт/лог, без переривання загального процесу імпорту.

**2.4. Збереження даних:**

- Збереження оброблених даних у базу даних (БД).
- Автогенерація таблиць: система повинна бути здатною динамічно створювати таблиці (або колекції) на основі проаналізованої структури файлу.
- Збереження метаданих (історія імпортів): дата старту та завершення завантаження, URL джерела, розмір файлу, кількість успішних рядків і кількість помилок.

**2.5. Інтерфейс користувача (Веб-кабінет):**

- **Дашборд:** Відображення зведеної статистики (останні імпорти, статус поточних фонових процесів, помилки).
- **Каталог наборів:** Інтерфейс для пошуку наборів безпосередньо через API data.gov.ua та їх швидкого додавання в розклад.
- **Управління завданнями:** Можливість ручного запуску/зупинки імпорту, а також завантаження локального файлу з комп'ютера (як альтернатива завантаженню за посиланням).
- **Перегляд результатів:** Табличний перегляд "сирих" завантажених даних з можливістю пошуку, сортування та експорту результатів (у CSV).
- **Журнал логів:** Детальний лог обробки кожного файлу з можливістю завантажити перелік рядків, що викликали помилку розбору (parsing error).

---

### 3. Нефункціональні вимоги та архітектура

**3.1. Архітектурні вимоги:**

- Архітектура: клієнт-серверна (SPA + REST/GraphQL API).
- Продуктивність: Додаток не повинен блокувати основний інтерфейс чи системні процеси під час імпорту гігабайтних архівів.
- Масштабованість: Модульна архітектура парсерів для можливості легкого додавання підтримки CSV або інших форматів у майбутньому.

**3.2. Рекомендований технологічний стек (може коригуватися):**

- **Backend:** Node.js (NestJS / Express).
- **База даних:** PostgreSQL (з підтримкою JSONB для збереження даних без жорсткої схеми).
- **Черги та фонові задачі:** Redis (bullMQ).
- **Frontend:** React.js / Vue.js + UI фреймворк (напр., TailwindCSS).
- **Контейнеризація:** Docker + Docker Compose для швидкого розгортання робочого середовища (БД+Backend+Frontend+Redis).

---

### 4. Вимоги до надійності та безпеки

**4.1. Відмовостійкість:**

- Наявність механізму повторних спроб (Retry Policy) у разі обриву мережевого з'єднання під час викачування файлу з data.gov.ua (напр., тайм-аути або 502 Bad Gateway State).

**4.2. Безпека:**

- Авторизація адміністраторів до панелі управління (Login/Password, JWT).
- Захист серверної частини від переповнень буфера (примусові ліміти під час стримінгу файлів).
- Екранування "брудних" даних (захист від SQL Injection).

---

### 5. Етапи розробки

1. **Етап 1: Проектування.** Затвердження стеку, підготовка структури БД, створення API-контрактів.
2. **Етап 2: Backend (Інтеграція та Парсинг).** Підключення до API data.gov.ua, розробка стрімінгових парсерів для XML/JSON/XLS, робота з брокером черг (модуль Workers).
3. **Етап 3: Frontend.** Розробка панелі адміністратора, моніторингу фонових процесів, таблиць перегляду результатів.
4. **Етап 4: Тестування.** Навантажувальне тестування на дійсних реєстрах з data.gov.ua (наприклад, "Єдиний державний реєстр юридичних осіб").
5. **Етап 5: Впровадження.** Розгортання (деплой) на серверах замовника за допомогою Docker, передача початкового коду та документації користувача.
